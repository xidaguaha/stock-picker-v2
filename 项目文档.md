# A股量化选股系统 v2 — 项目文档

**版本**: v6.10.2
**日期**: 2026-07-22
**架构**: 调度器常驻 + 6个子程序独立运行

---

## 一、项目简介

自动化 A股 量化选股系统。每日自动运行，通过 **量化因子快筛 + AI消息面深筛** 两层漏斗，筛选出最具潜力的个股，推送 Top3 到飞书群。

### 核心理念

```
量化因子（快，秒级）  →  筛出Top500  →  AI逐一分析（慢，分钟级）
        ↓                                      ↓
   技术/资金/估值/概念                    新闻/研报/催化剂/利空
        ↓                                      ↓
   综合得分 = 量化 × 0.92 + AI × 0.08  →  推送Top3到飞书
```

### 5.2 stock_ai_analysis.py（868行）— AI分析

#### 三源新闻抓取

| 来源 | 接口 | 内容 |
|------|------|------|
| 新浪财经 | 个股新闻页 | 主力来源，零鉴权 |

| 东财研报 | reportapi.eastmoney | 机构研报/评级 |

#### AI分析维度

行业定位、核心产品、下游应用、催化剂、利空因素、机构动态、行业景气度、AI评分(0-100)、投资逻辑、风险提示、关键词标签

#### 模型fallback链（全部免费）

```
deepseek-ai/DeepSeek-R1-0528-Qwen3-8B（主模型）
  → THUDM/GLM-Z1-9B-0414（9B推理）
  → THUDM/GLM-4-9B-0414（9B通用）
  → Qwen/Qwen2.5-7B-Instruct（7B通用）
  → Qwen/Qwen3-8B（兜底）
```

#### 缓存机制（按日期隔离）

**AI分析缓存** (`ai_analysis_cache.json`)：
- 今天有缓存 → 保留今天的，清除昨天的 → 直接用（不重新分析）
- 今天无缓存 → 清空所有 → 重新抓取分析
- 效果：今天不管打开多少次，只要跑过一次AI分析，后续都用缓存

**新闻磁盘缓存** (`news/YYYY-MM-DD.jsonl`)：
- 每天独立文件，今天只读今天的文件 → 天然按日期隔离

**新闻内存缓存** (`_mem`)：
- session级别，重启清空；key含日期，同一天内复用

### 5.14 ml_scorer.py（LightGBM机器学习打分）

**职责：** 用历史因子得分训练LightGBM模型，替代手工加权平均打分

**工作流程：**
```
历史因子得分(data/factor_scores/) + 涨跌幅
    ↓ 训练LightGBM回归模型
    ↓ 预测：因子得分 → 涨跌幅排名百分位(0-1)
    ↓ 归一化到0-100 → 综合得分
```

**核心逻辑：**
- 截面预测：同一天内所有股票的因子得分预测涨跌幅排名
- 自动回退：无模型或训练数据不足时，回退到手工加权平均
- 打分模式在日志中输出：`LightGBM ML` 或 `加权平均 (手工权重)`

**依赖：** `lightgbm>=4.0`, `scikit-learn>=1.3`

**模型存储：** `cache/lgb_model.pkl` + `cache/lgb_model_meta.json`

### 5.15 market_regime.py（市场环境自适应权重）

**职责：** 根据市场状态（牛/熊/震荡）动态调整因子权重

**判断指标：**
- 沪深300均线位置（MA20 vs MA60）
- 20日波动率（年化）
- 20日涨跌幅

**权重策略：**
| 环境 | 动量 | 估值 | 概念 | 质量 |
|------|------|------|------|------|
| 牛市 | 0.22↑ | 0.08↓ | 0.12↑ | 0.02↓ |
| 熊市 | 0.10↓ | 0.18↑ | 0.04↓ | 0.06↑ |
| 震荡 | 0.18 | 0.12 | 0.08 | 0.03 |

**集成方式：** `compute_composite_score` 在加权平均模式下自动检测市场环境并应用对应权重

### 5.16 行业分散选股

**逻辑：** 在TopN选取时，同行业最多2只，避免板块集中风险

**实现：** `select_diversified_topn(df, top_n=3, max_per_industry=2)`

### 5.17 sentiment_data.py（舆情数据模块）

**职责：** 接入akshare获取市场情绪指数和板块热度，作为额外因子纳入打分

**数据源：**
- A股新闻情绪指数（`index_news_sentiment_scope`）
- 概念板块热度排名（`stock_board_concept_name_em`）

**新增因子：**
- `情绪_市场`（0.02权重）：市场整体情绪分
- `情绪_板块`（0.02权重）：个股所属概念板块热度

### 5.18 llm_factor_miner.py（LLM因子挖掘模块）

**职责：** 用大模型定期扫描新闻，提取市场模式信号（政策/资金/轮动/催化剂）

**工作流程：**
```
热门概念列表 → LLM分析市场动态 → 结构化JSON信号
    → 缓存到 cache/llm_factor_signals.json（24小时有效）
    → 计算 LLM_信号 因子（0.02权重）
```

**信号维度：** 政策方向、资金流向、板块轮动、催化剂事件

**依赖：** 复用 stock_ai_analysis.py 的 call_ai 接口（SiliconFlow Qwen模型）

### 5.19 trading_calendar.py（动态交易日历）

**升级：** 从硬编码节假日表 → akshare动态获取 + 缓存 + 硬编码兜底

**数据源优先级：** 缓存(7天) → akshare → 硬编码(仅当年)

### 2026-07-14（v6.9.0 - 推送事故根因修复：__pycache__陈旧缓存+空DataFrame保护）

**事故：7月14日9:21竞价推送完全失败（连续3次exit code=1），飞书无推送**

**根因分析**：
1. **__pycache__陈旧缓存**（根本原因）：7月13日修改了`stock_picker.py`的is_auction逻辑（竞价时段放宽PE/市值过滤），但Python加载了`__pycache__`中旧的`.pyc`文件（编译自修改前的代码）。旧代码在9:21竞价时段对4991只股票应用PE>0和市值>=20亿过滤，而竞价时段PE/市值数据为0/NaN，导致全部4991只被过滤（清洗后0只）
2. **空DataFrame崩溃**：清洗后0只股票时，`format_output()`和`df['综合得分'].iloc[0]`抛IndexError，导致exit code=1

**修复1（CRITICAL）：scheduler.py添加__pycache__自动清理**
- 在`run_subprocess()`函数中添加`_clear_pycache()`调用，每次运行子脚本前清理`共用模块/`、脚本目录、项目根目录下的所有`__pycache__`
- 确保Python从`.py`源码重新编译，永远不使用陈旧`.pyc`缓存
- **影响**：每次运行增加~1-2秒编译时间（stock_picker.py 180KB），对30分钟+的AI分析流程可忽略

**修复2（CRITICAL）：竞价量化推送/run.py添加空DataFrame保护**
- 清洗后`len(df)==0`时graceful退出，保存空快照（防止scheduler反复重试），不再crash
- **影响**：即使数据源异常导致全量过滤，也不会exit code=1

**验证**：
- 13:53手动推送成功（清洗后2679只，飞书推送成功✓）
- daemon已重启（PID 40976），加载新scheduler.py代码
- is_auction逻辑在9:21正确放宽过滤（in_session_now=False + is_auction=True）

- 修复：改为截止时间计算 `deadline = time.time() + total_seconds`，循环检查剩余时间
- 用户此前反馈 Windows 从未休眠，此修复与 Windows 休眠无关

**问题2：腾讯/同花顺数据源长期失败**
- 腾讯 K 线 API：`https://web.ifzq.gtimg.cn/appstock/app/fqkline/get` 经实际测试完全正常，返回完整 `qfqday` 数据
- 腾讯在 `kline_fetcher.py` 中原排第 6 位（baostock→xueqiu→akshare→ths→sina→tencent），baostock 通常成功导致腾讯几乎不被调用
- 修复：将腾讯优先级提升至第 1 位（腾讯→新浪→baostock→xueqiu→akshare→ths）
- 同花顺失败根因：环境中 **akshare 未安装**（`ModuleNotFoundError`），且 `THS_COOKIE_V` 未配置
- 结论：同花顺接口本身未关闭，需用户自行 `pip install akshare` 并在 `config.json` 的 `credentials.ths_cookie_v` 中配置 cookie

**问题3：执行耗时超过1小时**
- 根因1：`compute_factors` 中 Baostock ROE 查询全量 1000+ 只股票，每只 ~1.5s，总耗时约 28 分钟
- 修复1：限制 ROE 查询只查市值 Top500（与选股池实际规模匹配）
- 根因2：技术因子计算需为每只股票拉取 60 日 K 线，5000+ 只股票网络请求密集
- 修复2：新增 **盘前预热机制**
  - 新建 `盘前预热.py`：提前缓存股票列表、概念映射、60日K线、Baostock估值/行业/ROE
  - 新建 `盘前预热.bat`：支持手动执行
  - `scheduler.py` 新增 8:00-8:10 "盘前数据预热" 阶段，自动调度
  - `stock_picker.py` 股票列表缓存有效期从自然日改为 24 小时，支持晚上预热次日复用
  - 预热后竞价阶段只需获取实时行情，执行时间预计从 60+ 分钟降至 5 分钟以内

**其他修复**：
- `竞价量化推送/run.py` 增加本地 `log()`/`warn()`/`error()` 定义，修复线程中 `name 'log' is not defined`
